# Architectural Review
> Generated by /ts-code-viewer — 2026-06-03

## Structural Summary

The implementation is a 4-layer Fastify API: **HTTP (routes/handlers/schemas) → Domain (controllers/SessionStore/QuoteEngine) → Data Model (types) → Infra (WebError/errorHandler)**. Dependency direction is strictly top-down with one intentional cross-layer import (`OnboardingWebError extends WebError`). No cycles. No violations.

The domain layer is Fastify-free — controllers are pure functions, SessionStore and QuoteEngine have no HTTP knowledge. The error hierarchy is the most notable design decision: 4 typed domain errors map deterministically to 4 HTTP status codes in a single `mapDomainError` function.

---

## Review Priorities

### High
- **`submitQuote` calls `store.get()` twice** — once to check step, once after `update()` to get the fresh session for QuoteEngine. The second `get()` could be replaced by returning the updated session from `update()` (which already does so). Low risk but worth cleaning.

### Medium
- **`STEP_ORDER` is declared in two places** — `SessionStore.ts` and `controllers.ts` both define their own `STEP_ORDER` array. Should live in `types.ts` or a constants module and be imported by both. If a new step is added, both must be updated.
- **`handlers.ts` constructs singletons at module level** — `const store = new SessionStore()` at the top of `handlers.ts` means the store is shared across all tests that import `buildApp()`. Tests are currently isolated because each test builds a fresh app... but `buildApp()` re-imports the same module-level singleton. This is a latent test-isolation risk.

### Low
- **`StepOrderError` base class still exported from `errors.ts`** — was kept for backwards compatibility but is now unused. Safe to remove.
- **`carRepository.getAllCars()` is called on every `submitProfile`** — fine at this scale, but if the catalogue were large or async this would be the first thing to cache or lazy-load.

## Likely Risks

### Correctness
- **TTL check uses `Date.now()` vs `createdAt`** — correct, but `createdAt` is set at `new Date()` construction time. If the system clock skews (e.g. VM suspend/resume), sessions could expire sooner or later than expected. Acceptable for interview scale.
- **`store.update()` after `advanceStep()` is not atomic** — `submitQuote` does: update → get → calculate → update → advanceStep. Between the second `update` and `advanceStep`, a concurrent call could observe intermediate state. Not a real risk in single-threaded Node.js, but worth naming in an interview.

### Maintainability
- **`mapDomainError` must be kept in sync with domain error classes** — adding a new domain error without updating `mapDomainError` causes a silent re-throw → 500. A TypeScript exhaustive-check pattern would catch this at compile time.

## Questions for the Implementer

1. **`STEP_ORDER` is duplicated** — why not extract it to `types.ts` or a `constants.ts` and import from both `SessionStore` and `controllers`?
2. **The `store` singleton in `handlers.ts`** — if `buildApp()` is called twice in tests, do both instances share the same store? What's the test-isolation strategy?
3. **`submitQuote` calls `store.get()` twice** — could `update()` return the updated session directly (it already does), avoiding the second round-trip?
4. **`mapDomainError` is not exhaustive** — if a new domain error is added, it silently falls through to a 500. Would you use a discriminated union or a switch with a compile-time exhaustiveness check here?
5. **What's the first thing you'd change to support 10× load?** _(Expected answer: extract `SessionStore` behind a `SessionRepository` interface and inject it; swap `Map` for Redis; move singleton construction to `buildApp()`.)_

## Suggested Refactors

### 1. Extract `STEP_ORDER` to `types.ts` (5 min)
```ts
// types.ts
export const STEP_ORDER: SessionStep[] = [
  SessionStep.STARTED, SessionStep.PROFILED, SessionStep.QUOTED, SessionStep.BOUND,
];
```
Import in both `SessionStore.ts` and `controllers.ts`. Single source of truth.

### 2. Remove `store.get()` second call in `submitQuote` (2 min)
```ts
// controllers.ts — current
store.update(sessionId, { selectedCarIds: body.carIds });
const updatedSession = store.get(sessionId);   // ← unnecessary
const quote = engine.calculate(updatedSession);

// simplified
const withCarIds = store.update(sessionId, { selectedCarIds: body.carIds });
const quote = engine.calculate(withCarIds);
```

### 3. Move singleton construction to `buildApp()` (10 min — scale-prep)
```ts
// app.ts
const store = new SessionStore();
const engine = new QuoteEngine({ perCarFactors: [new AgeFactor()], totalFactors: [new CarCountFactor()] });
app.decorate('store', store);
app.decorate('engine', engine);
```
Then handlers receive them via `request.server.store` / `request.server.engine`. Enables per-test store injection, and is the first step toward Redis swap.
