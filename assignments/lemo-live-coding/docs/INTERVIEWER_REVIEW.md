## 1. Core Clean Code Principles — 9/10

const ELIGIBLE_CARS: CarOption[] = CAR_CATALOGUE.filter(isEligible);

## 2. Architectural & Design Patterns — 9/10

Clean 4-layer architecture: `routes → handlers → controllers → domain`. No violations detected by dependency-cruiser after refactors.

`makeOnboardingHandlers()` factory correctly isolates `SessionStore` and `QuoteEngine` construction in `routes.ts` — each `buildApp()` call gets a fresh store. Domain has zero Fastify imports.

**Optimization** — `submitQuote` still does two store writes (update + advanceStep). Could be reduced to one by adding an optional patch parameter to `advanceStep`, making the step transition atomic with data. Minor at this scale, worth naming:

Not worth implementing now — current approach is clear and correct.

## 3. TypeScript Idioms & Modern Practices — 8/10

No `any`, no unsafe casts, strict mode honored throughout.

**Warning** — `Session` interface has 5 optional fields forming an implicit state machine. The compiler allows constructing a `BOUND` session with no `quote`:

```ts
// currently valid — compiler won't catch this
const bad: Session = { sessionId: 'x', step: SessionStep.BOUND, createdAt: new Date() };
```

**Fix** — discriminated union makes illegal states unrepresentable:
```ts
type StartedSession  = { step: SessionStep.STARTED; sessionId: string; createdAt: Date };
type ProfiledSession = StartedSession & { step: SessionStep.PROFILED; profile: ProfileBody; eligibleCars: CarOption[] };
type QuotedSession   = ProfiledSession & { step: SessionStep.QUOTED; selectedCarIds: string[]; quote: Quote };
type BoundSession    = QuotedSession  & { step: SessionStep.BOUND; boundAt: Date };
type Session = StartedSession | ProfiledSession | QuotedSession | BoundSession;
```

This is a non-trivial refactor — the right "if I had more time" answer.

**Style** — `STEP_ORDER` could use `as const satisfies SessionStep[]` for stricter tuple inference:
```ts
export const STEP_ORDER = [
  SessionStep.STARTED, SessionStep.PROFILED, SessionStep.QUOTED, SessionStep.BOUND,
] as const satisfies SessionStep[];
```

**Style** — `makeOnboardingHandlers` return type is inferred. Explicit return type would aid discoverability for a public factory function.

## 4. Bugs, Edge Cases & Robustness — 8/10

**Warning** — `submitQuote` does not validate that `body.carIds` are a subset of `session.eligibleCars`. A client submitting an ineligible `carId` gets a `total: 0` quote silently (the `if (!car) continue` path in `QuoteEngine`).

```ts
// fix in submitQuote — validate before computing
const eligibleIds = new Set(session.eligibleCars?.map((c) => c.carId) ?? []);
const invalidIds = body.carIds.filter((id) => !eligibleIds.has(id));
if (invalidIds.length > 0) {
  throw new StepPrerequisiteError(`Car IDs not eligible: ${invalidIds.join(', ')}`);
}
```

**Style** — `SessionStore.update()` does not check TTL, unlike `get()`. Controllers always call `get()` first so this is currently safe, but it's a latent inconsistency — a future controller that calls `update()` directly could mutate an expired session.

## 9. Scalability & Scale-Out Readiness — 8/10

Scale limits are named in `DESIGN.md` and the implementation stays honest to them. Core logic is pure, I/O is at boundaries, no O(N²) in hot paths after the `Map` fix.

**Warning** — `SessionStore` in-memory `Map` grows unbounded. Expired sessions linger until accessed — a client that creates sessions and never calls them again leaks memory indefinitely. Named in `DESIGN.md` but not mitigated in code. Minimum fix: remove expired entry on access in `get()`:

```ts
if (Date.now() - session.createdAt.getTime() > this.ttlMs) {
  this.sessions.delete(sessionId); // evict on access
  throw new SessionExpiredError(sessionId);
}
```

**Optimization** — `submitProfile` calls `getAllCars().filter(isEligible)` on every request. Pre-compute once.

**Note** — `SessionStore` is the only real scaling bottleneck. The `makeOnboardingHandlers` injection point makes the swap to Redis clean: pass a `SessionRepository` interface instead of `SessionStore`, inject a Redis-backed implementation in `routes.ts` — zero handler or controller changes.

## Verdict

**Strong senior-level TypeScript engineering.** The architecture is clean and consistently enforced across all layers. Error hierarchy is well-designed — domain errors are HTTP-free, mapping is centralised in `mapDomainError`, `OnboardingWebError extends WebError` from the base server is the right abstraction. The `makeOnboardingHandlers` factory correctly solves test isolation. O(N×M) replaced with O(N+M) Map lookup. Named constants throughout, no cycles, no `any`.

The two remaining discussion-worthy gaps: (1) `Session` as a flat interface with optional fields vs. a discriminated union — the right "if I had more time" answer; (2) unbounded `SessionStore` memory growth — named and documented, minimum fix is one-line eviction on `get()`.

## Scale-up Q&A

**Q:** Sessions grow indefinitely in memory — what's the minimum fix and what's the real fix?
**A:** Minimum: evict the session entry in `get()` before throwing `SessionExpiredError` — one line. Real fix: extract a `SessionRepository` interface, inject it via `makeOnboardingHandlers`, swap the `Map` for a Redis-backed implementation with native key TTL. The injection point is already in place — zero handler changes needed.

**Q:** `submitProfile` filters the car catalogue on every request — how would you fix it?
**A:** Pre-compute `ELIGIBLE_CARS` at module load in `carRepository.ts`. It's a static array — filtering it on every request is pure waste. If the catalogue became dynamic, push the filter to the query layer: `SELECT * FROM cars WHERE year >= 2008 AND value < 150000`.

**Q:** The `Session` type allows constructing a `BOUND` session with no `quote` — how would you fix that?
**A:** Replace the flat interface with a discriminated union: `type Session = StartedSession | ProfiledSession | QuotedSession | BoundSession`, where each variant only has required fields for that step. The compiler enforces that a `BOUND` session always has `boundAt` and `quote` — no runtime guards needed.

**Q:** `submitQuote` does two store writes — why, and how would you reduce it to one?
**A:** Currently: `update({selectedCarIds, quote})` then `advanceStep(QUOTED)`. Could be one write if `advanceStep` accepted an optional data patch, making the transition atomic. At this scale it's fine; against a real DB it's two round-trips per quote request — worth fixing when switching to a persistent store.

**Q:** What's the first thing you'd change to support 10× the current load?
**A:** Extract `SessionRepository` interface, inject it in `routes.ts`, swap `new SessionStore()` for a Redis-backed implementation. Sessions become stateless from the Node process's perspective — any number of processes can serve requests. The injection point is already in `makeOnboardingHandlers` — it's a one-file change.
