## Structural Summary

- **10 modules** across 5 layers: Entrypoint → App Factory → Infrastructure → HTTP → Domain
- **Dependency direction:** strictly top-down, zero violations, zero cycles
- **Center of gravity:** `app.ts` (in-degree 1, out-degree 4) — it is the composition root that wires everything together
- **Leaf modules:** `config.ts`, `schemas.ts`, `controllers.ts`, `plugins/swagger.ts`, `errors/errorHandler.ts` — all have out-degree 0; all are safe to replace independently
- **Test strategy:** all HTTP behavior tested via `fastify.inject()` — no real port, no mocking of Fastify internals; domain logic tested in pure unit tests without any HTTP layer

## Review Priorities

### High (must understand for interview)
- **`buildApp()` factory pattern** — separates app construction from network listening; enables `inject()`-based testing without ports. Interviewers will ask why `server.ts` and `app.ts` are split.
- **Schema-driven validation + OpenAPI** — `schemas.ts` is the single source of truth for both AJV validation and the generated OpenAPI spec. No separate validation library needed.
- **Layer boundary in `handlers.ts`** — this is where Fastify types stop. `controllers.ts` has zero framework imports. Be ready to explain why.

### Medium (worth knowing)
- **`errorHandler.ts` only catches non-validation errors** — Fastify's AJV pipeline handles 400s before `setErrorHandler` is invoked. Custom handler only fires for 5xx.
- **`additionalProperties: false` strips (not rejects) extra fields** — AJV's `removeAdditional` is Fastify's default. The test documents this explicitly.
- **`pino-pretty` transport is dev-only** — enabled only when `NODE_ENV=development`; tests run without it to avoid the transport dependency.

### Low (nice to mention)
- **`genReqId` uses `crypto.randomUUID()`** — built-in Node.js, no extra dependency. Produces a UUID v4 per request for correlation.
- **`tsconfig.test.json`** extends the main tsconfig with `rootDir: "."` so `tests/` can import from `src/` without a `rootDir` conflict.

## Likely Risks

### Correctness risks
- **None in the current placeholder implementation.** `postLemo()` is a pure echo with no logic that could fail.
- **Interview day:** once real business logic is added to `controllers.ts`, error handling inside `postLemo()` needs a decision — throw a typed error (caught by `errorHandler`) or return a discriminated union. The current structure supports either, but pick one and stick to it.

### Maintainability risks
- **`handlers.ts` is a thin pass-through.** If interview logic grows complex, resist the temptation to put logic in handlers — keep pushing it into controllers.
- **`config.ts` has no validation** — `parseInt` on a non-numeric `PORT` produces `NaN`, which Fastify will reject at listen time with a confusing error. Acceptable for prep; add a guard if the real prompt involves config parsing.
- **`getLemo()` returns `void`** — correct for the placeholder, but if the GET route needs to return data on interview day, both the controller and the schema need updating together. The interview-day checklist in `DESIGN.md` flags this.

## Questions for the Implementer

1. **Why is `buildApp()` async?** — Because `@fastify/swagger` and `@fastify/swagger-ui` are async Fastify plugins that must be awaited before routes are registered.

2. **Why does `registerErrorHandler` not re-throw validation errors?** — Because Fastify's AJV pipeline sends 400s directly without invoking `setErrorHandler`. The custom handler only fires for errors that escape route handlers — typically 5xx.

3. **What happens if `POST /lemo` receives `{ name: 1, value: "ok" }`?** — AJV rejects it with a 400 because `name` must be a string. This is enforced by the schema, not by any code in handlers or controllers.

4. **Could you swap Fastify for Express?** — Yes. Only `app.ts`, `routes.ts`, `handlers.ts`, and the plugins would change. `controllers.ts`, `schemas.ts` (the interfaces), and `config.ts` are framework-agnostic.

5. **Why are tests in `tests/` rather than `src/__tests__/`?** — The assignment spec required a specific file layout with `tests/` at the root. The `tsconfig.test.json` handles the `rootDir` conflict this creates.

## Suggested Refactors

All items below are **"worth discussing"**, not **"must fix"**. The current code is correct and clean for the assignment scope.

- **Add `PORT` validation in `config.ts`** — `if (isNaN(config.port)) throw new Error('PORT must be a number')` — one line, prevents a confusing Fastify startup error.
- **Type the `errorHandler` more tightly** — `FastifyError` already has `statusCode?: number`; the current `?? 500` fallback is correct but could be made explicit with a comment for interview explanation.
- **Consider naming the OpenAPI operation IDs** — add `operationId: 'getLemo'` / `operationId: 'postLemo'` to the schemas for cleaner generated client code. Low priority for prep, high value for real APIs.
