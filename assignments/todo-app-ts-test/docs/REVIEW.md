## Structural Summary

The codebase has a clean 3-layer architecture: Data Model (`types.ts`) ← Domain Logic (`taskManager.ts`) ← Public API (`index.ts`) ← Tests. Dependency direction is strictly inward with no back-edges, no sideways imports, and no cycles. The center of gravity is `types.ts` (highest centrality), which is appropriate — it defines the core domain contract that all other layers depend on. The structure is minimal, readable, and would be easy to extend.

## Review Priorities

**High**

- **`completed` is mutable on the public `Task` interface** (`types.ts:4`)
  Any consumer holding a `Task` reference can write `task.completed = false` directly, bypassing `TaskManager`. This means `TaskManager` doesn't own the state it manages.
  Fix: mark `completed` as `readonly` in the interface and use a spread copy in `completeTask`.

- **`completeTask` mutates the live object in the Map** (`taskManager.ts:27`)
  `task.completed = true` modifies the object stored inside the `Map`. The returned reference is the same object — callers who held the return value from `addTask` see it change silently.
  Fix: `const updated = { ...task, completed: true }; this.tasks.set(id, updated); return updated;`

**Medium**

- **`new Date()` is hardcoded — clock is not injectable** (`taskManager.ts:16`)
  `createdAt` cannot be controlled in tests without real-time coupling. The test brackets `before`/`after` are fragile on slow CI.
  Fix: accept an optional `clock: () => Date` parameter in the constructor, defaulting to `() => new Date()`.

- **Duplicated "not found" error message template** (`taskManager.ts:25`, `taskManager.ts:33`)
  Both `completeTask` and `deleteTask` construct the same string independently. A format change requires two edits.
  Fix: add a `static notFound(id: number): TaskError` factory to `TaskError`.

**Low**

- **`deleteTask` does two Map lookups** (`taskManager.ts:32–35`)
  `has(id)` + `delete(id)` — `Map.delete` returns `false` if the key didn't exist, so the `has` check is redundant.
  Fix: `if (!this.tasks.delete(id)) throw TaskError.notFound(id);`

- **`noUncheckedIndexedAccess` is disabled** (`tsconfig.json:7`)
  Unguarded array accesses (`tasks[0].title`) compile without error. Enabling it would catch these across the project.
  Fix: set `"noUncheckedIndexedAccess": true`.

- **ESLint `explicit-function-return-type` is `warn` not `error`** (`.eslintrc.json`)
  The project standards call explicit return types non-negotiable — the lint rule should enforce this, not just warn.
  Fix: change to `"error"`.

## Likely Risks

**Correctness risks**
- Mutable `completed` field allows callers to corrupt `TaskManager` state from outside. This is the most serious issue — it breaks the encapsulation guarantee of the class.
- A caller holding the reference returned by `addTask` will see `completed` flip to `true` after `completeTask` is called, without any explicit call on their side. This is surprising and hard to debug.

**Maintainability risks**
- Non-injectable clock makes `createdAt`-related tests timing-sensitive. On fast machines this never fails; on slow CI it occasionally does.
- Duplicated error message strings will drift if the format changes — one call site gets updated, the other doesn't.
- Tests importing from `../index` (now fixed) — was a refactor risk, now resolved.

## Questions for the Implementer

1. `Task.completed` is not `readonly` — what prevents a caller from setting it directly without going through `completeTask`? Is that intentional?
2. `completeTask` returns the same object reference that's stored in the Map. What happens if the caller mutates the returned object?
3. How would you test that `createdAt` is set to exactly the right time in a deterministic way?
4. `deleteTask` calls `has` then `delete` — do you know that `Map.delete` already tells you whether the key existed?
5. If you needed to add a `updateTitle(id, newTitle)` method tomorrow, what would change in the current architecture?

## Suggested Refactors

Proportionate to the assignment size — listed from highest to lowest impact:

1. **Make `Task.completed` readonly + use spread copy in `completeTask`**
   One-line interface change + 3-line method rewrite. Fixes the encapsulation bug and eliminates mutation-through-reference.

2. **Inject the clock into `TaskManager`**
   Add `constructor(clock: () => Date = () => new Date())`. Makes `createdAt` deterministically testable.

3. **Add `TaskError.notFound(id)` static factory**
   Eliminates the duplicated string template and makes both call sites one-liners.

4. **Collapse `has` + `delete` into a single `delete` call in `deleteTask`**
   Two lines become one. Demonstrates Map API fluency.

5. **Enable `noUncheckedIndexedAccess: true` in tsconfig**
   Zero code changes required if the existing `?.` guards are already in place — just tightens the type checker.
